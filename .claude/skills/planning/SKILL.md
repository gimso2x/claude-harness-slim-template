# Planning Skill

## 목적
큰 작업을 바로 구현하지 않고 범위, 단계, 리스크를 먼저 정리한다.

## 언제 사용하나
- 파일 여러 개를 수정할 때
- 새 기능 / 구조 변경 / 리팩터링일 때
- 먼저 접근법 합의가 필요할 때
- 작업 중 컨텍스트가 길어질 가능성이 클 때

## 출력 규칙
- 작은 작업: 대화 안에서 짧은 계획
- 중간 이상 작업: `docs/plans/YYYY-MM-DD-{slug}.md` 생성

## 절차
1. 배경과 문제를 정리한다.
2. 목표와 범위를 구분한다.
3. 변경 대상 파일/모듈을 예상한다.
4. 구현 단계를 작은 단위로 쪼갠다.
5. 테스트/검증 방법을 정한다.
6. 리스크/미확정 사항을 적는다.
7. 사용자에게 요약을 보여주고 진행한다.

## 템플릿
```md
# {작업명} 계획

- 날짜: YYYY-MM-DD
- 상태: draft | approved | in-progress | done

## 배경

## 목표

## 범위
### 포함
### 제외

## 대상 파일 / 모듈

## 구현 단계
1.
2.
3.

## 테스트 / 검증
- [ ]
- [ ]

## 리스크 / 논의 필요 사항
- 
```
